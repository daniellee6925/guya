## 1. What this migration does

The letter prefixes are backwards. Today:

- `T-NNN` lives in `tasks/proposals/` but reads like "Task" → it is actually a **proposal**.
- `P-NNN` lives in `tasks/tasks/` but reads like "Proposal" → it is actually a **task**.

Target convention: **`T` = Task** (active work), **`P` = Proposal** (staged idea). Directories stay where they are; only the *letter* and all references swap.

Inventory (small, recent — all within the last ~2 weeks):
- Proposals (currently `T-`): `T-001`…`T-011` (11) → become `P-001`…`P-011`
- Tasks (currently `P-`): `P-001`…`P-005` (5) → become `T-001`…`T-005`

## 3. The two findings that reshape the task-as-written

**(a) "Single atomic commit" is impossible — it spans three repos.**
- `~/Desktop/constantia` — task/proposal files, hooks, schema docs
- `~/Desktop/telos` — MCP server code + tick prompts (the prefix-minting logic)
- `~/Desktop/guya` — STATUS.md, ARCHITECTURE.md, ADR
Atomicity is **per-repo**. Consistency across repos is achieved by a **freeze window**, not a single commit.

**(b) Corruption window on a running system.**
The Telos MCP server (`shared/telos-tools/mcp-server.ts` + `helpers.ts`) is **baked into the nanoclaw container** and run by `com.nanoclaw.plist` ticks; `constantia-sync.sh` pushes from the host every 5s. If Constantia's files flip while the **old** container is still live, the next tick's `nextProposalId()` (still minting `T-`) would write a `T-` proposal into a world where `T-` now means a task → silent ID collision. **The running system must be frozen during cut-over and the container rebuilt before resume.**
