# Constantia — Core Beliefs

> Last updated: 2026-04-20

## What This Program Is

Constantia is a living, evidence-backed model of Daniel — his trajectory, commitments, strengths, weaknesses, and active work — that serves as the single source of truth for both Guya and Telos. It is the shared memory that makes two independent agents into one coherent system.

## The Differentiator

Most multi-agent systems either share nothing (divergence) or share everything (coupling). Constantia sits between: a structured repo with clear ownership boundaries per file, where each agent writes to its own surfaces and reads from the whole. The divergence problem disappears because truth lives in one place. The coupling problem disappears because agents don't need each other to function — they need Constantia.

## Core Design Principles

### 1. Single source of truth — if it's not in Constantia, it didn't happen

**The principle:** Constantia is the authoritative record of Daniel's commitments, progress, growth, and goals. Guya's local state and Telos's local state are caches and operational memory. Constantia is the ledger.

**Why it matters:** The entire reason this system exists is to eliminate divergence between two agents. If either agent maintains authoritative state outside Constantia, the divergence problem returns through the back door.

**What violation looks like:** Telos grading Daniel based on observations stored only in its local log. Guya tracking growth in `~/.claude/guya/growth-tracker.md` without syncing claims to Constantia's profile. A fact about Daniel that exists in one agent's memory but not in Constantia.

**Decision filter:** If Telos and Guya disagree about a fact, which source wins? If the answer isn't "Constantia," the architecture is broken.

---

### 2. Evidence-grounded claims — no source, no claim

**The principle:** Every assessment in `evidence/` and every claim in `profile/` must point to a specific log entry, commit SHA, or artifact. Claims without sources are opinions, not knowledge.

**Why it matters:** This is what makes Constantia a model of Daniel rather than a collection of vibes. When Telos says "Daniel is weak at X," the evidence chain is traceable: profile claim → evidence entry → log entry → artifact. Daniel can audit any assessment end-to-end.

**What violation looks like:** A profile strength entry without a `source:` field. An evidence assessment that says "generally observed" without pointing to a specific date and event. A grade that cites "overall impression" instead of a specific artifact and criterion.

**Decision filter:** Can Daniel trace this claim back to a specific artifact he produced on a specific date? If no, the claim can't be written.

---

### 3. Clear ownership — no shared-write files

**The principle:** Every file in Constantia has exactly one write-owner. Guya writes `log/` entries and updates task status. Telos writes `evidence/`, `profile/`, `goals/`, and task assignments/grades. Manifests are generated by hook, not written by agents. No file is written by both.

**Why it matters:** Two agents writing the same file means merge conflicts, format drift, and silent overwrites. File-level ownership eliminates all three. If you never have two writers, you never have a conflict.

**What violation looks like:** Guya editing an evidence entry. Telos modifying a log entry Guya wrote. Either agent manually editing a MANIFEST.md file. A new file type added without declaring its owner.

**Decision filter:** Who writes this file? If the answer is "both," redesign.

---

### 4. One entity per file — writes can't corrupt neighbors

**The principle:** Tasks, evidence entries, and log entries are individual files. No giant YAML arrays, no append-to-shared-file patterns. Creating TASK-025 cannot corrupt TASK-024.

**Why it matters:** Agents are imperfect writers. An LLM that appends to a 200-line shared file risks malforming the whole file. An LLM that writes a single self-contained file can only corrupt that one entity. Blast radius is bounded by design.

**What violation looks like:** Multiple tasks stored in a single `tasks.yaml`. Evidence entries appended to a shared `evidence/strengths.md` file. Log entries that append to a growing monthly file instead of being daily files.

**Decision filter:** If this write fails or produces malformed output, how many other entities are affected? If more than zero, split.

---

### 5. Manifest-indexed — agents read one file to navigate the repo

**The principle:** Each directory has a MANIFEST.md that indexes all entries with key metadata (frontmatter fields). Agents read the manifest first, then drill into specific files. Manifests are auto-generated by pre-commit hook, never hand-written.

**Why it matters:** An agent can't efficiently scan 200 files. The manifest is the "table of contents" that makes navigation O(1) instead of O(n). Because it's auto-generated, it never goes stale. Because agents never write it, it never gets corrupted.

**What violation looks like:** An agent scanning all files in a directory instead of reading the manifest. An agent manually editing a manifest. A manifest that's out of sync with actual file contents (hook not running).

**Decision filter:** Can this lookup be answered by reading the manifest alone? If yes, don't read individual files.

---

### 6. Reliability over features — data survives bad ticks

**The principle:** Constantia is a git repo with a remote. All writes go through git commits. Pre-commit hooks validate frontmatter schema. Auto-push on commit. The repo is never in an invalid state after a commit completes.

**Why it matters:** Constantia is the most valuable artifact in the system — months of accumulated evidence, assessments, and growth data. A corrupted Constantia is worse than no Constantia, because both agents would operate on wrong information. Every write path must be safe.

**What violation looks like:** Writing directly to files without committing. Skipping schema validation to save time. A state where `git status` shows uncommitted changes for more than minutes. No remote backup.

**Decision filter:** If power cuts during this operation, what's the worst state Constantia can be in? If the answer is "corrupt" or "inconsistent," add a safety boundary.

---

## What This Is Not

- **Not a communication channel.** Agents don't send messages through Constantia. They write state and read state. The "communication" is implicit in state changes, not explicit in messages.
- **Not a task manager.** Tasks exist here because they're evidence of what Daniel committed to and how well he executed. The task format serves accountability, not project management ergonomics.
- **Not a database.** It's a git repo with markdown files. The query capability is grep + manifest indexes. If you need joins or aggregations, you're over-engineering.
- **Not agent-specific.** Constantia doesn't know or care whether Guya or Telos is reading it. It's a data repo. The agents are clients. If a third agent appeared tomorrow, it could read Constantia with zero changes.
